# Vault-Native Skill Commands

This document specifies Vulcan's callable automation model for Agent Skills-compatible skill packages.

Vulcan uses the official Agent Skills directory shape as the portable packaging format:

```text
.agents/skills/<skill-name>/
  SKILL.md          # Required: metadata + instructions
  scripts/          # Optional: executable commands
  references/       # Optional: detailed docs loaded on demand
  assets/           # Optional: templates, schemas, static resources
```

A skill teaches an agent or human how to perform a workflow. A **skill command** is a named executable entrypoint inside that skill. Vulcan can project trusted skill commands into its shared tool registry so they are callable from CLI, MCP, `describe`, internal JS, and future assistant runtimes without inventing a second package format.

## Goals

- Stay compatible with Agent Skills by treating `SKILL.md` as the package root.
- Keep skills vault-native: reviewed, versioned, linked, and shipped with the vault.
- Reuse the existing QuickJS runtime, trust model, permission profiles, and sandbox model.
- Make command discovery static and cheap: list and describe commands from `SKILL.md` metadata only, with no code execution during discovery.
- Project commands into the same registry as built-in Vulcan tools so CLI, MCP, `describe`, and internal JS do not drift.
- Preserve the rule that vault mutations go through Vulcan semantics: permission checks, diagnostics, checkpoints, diffs, and audit records.
- Support host process execution deliberately and explicitly rather than smuggling it behind unrestricted script execution.

## Non-goals for the initial version

- A separate `.agents/tools/<name>/TOOL.md` package format as the primary model.
- Dynamic schemas generated by running user code at discovery time.
- Long-lived in-memory command daemons or background workers.
- Arbitrary dependency loading from npm, pip, or remote URLs.
- Full WASM runtime support in the first implementation.

## Core model

Vulcan distinguishes four automation surfaces:

- Skills teach workflows with Markdown instructions, examples, references, and companion commands.
- Skill commands are directly callable request/response entrypoints declared by a skill.
- Plugins react to Vulcan lifecycle events.
- `vulcan run` executes one-off or exploratory scripts that are not yet projected as commands.

If executable behavior should be discoverable and callable through CLI, MCP, `describe`, and the JS API, it should be declared as a skill command in `SKILL.md` metadata. If the behavior is still experimental or personal glue, start with `vulcan run`.

## Skill layout

Recommended layout:

```text
.agents/skills/
  daily-review/
    SKILL.md
    scripts/
      prepare-day.js
      process-inbox.js
      extract-commitments.js
    schemas/
      prepare-day.input.schema.json
      prepare-day.output.schema.json
    references/
      task-policy.md
    assets/
      daily-note-template.md
```

`schemas/` is not part of the core Agent Skills directory set, but additional directories are allowed. Vulcan treats it as a conventional place for JSON Schemas used by skill commands.

## `SKILL.md` metadata

The official skill frontmatter fields remain the public compatibility surface:

- `name`
- `description`
- `license`
- `compatibility`
- `allowed-tools`
- `metadata`

Vulcan-specific command metadata lives under `metadata.vulcan`.

Example:

```markdown
---
name: daily-review
description: Prepare daily briefings, process inboxes, extract commitments, and maintain daily notes. Use when planning the day, reviewing tasks, or cleaning up captured notes.
license: MIT
compatibility: Requires Vulcan for vault access. Commands use the Vulcan skill-command runtime.
allowed-tools: vulcan
metadata:
  vulcan:
    commands:
      prepare-day:
        script: scripts/prepare-day.js
        description: Prepare a daily briefing from calendar, tasks, and recent notes.
        input_schema: schemas/prepare-day.input.schema.json
        output_schema: schemas/prepare-day.output.schema.json
        sandbox: strict
        permission_profile: readonly
        packs: [custom, life-assistant]
        read_only: false
        destructive: false
        permissions:
          - vault.read
          - vault.write:Daily
          - integration.calendar.read
        expose:
          cli: true
          mcp: true
          describe: true
      process-inbox:
        script: scripts/process-inbox.js
        description: Classify captured inbox items and propose filing actions.
        input_schema: schemas/process-inbox.input.schema.json
        sandbox: strict
        permission_profile: assistant-edit
        packs: [custom, life-assistant]
        permissions:
          - vault.read
          - vault.write:Inbox
          - vault.write:Tasks
        expose:
          cli: true
          mcp: true
          describe: true
---

# Daily Review

Use this skill when preparing a day, reviewing tasks, processing captured inbox items, or extracting commitments from recent notes and conversations.

## Available commands

- `prepare-day`: prepares a daily briefing.
- `process-inbox`: classifies and files captured inbox items.

## Workflow

1. Run `prepare-day` with `dry_run: true`.
2. Review the proposed changes.
3. Run again with approval if the changes are correct.
```

## Command metadata fields

Each entry under `metadata.vulcan.commands` declares one projected command.

Required fields:

- `script`: path relative to the skill root
- `description`: one-line summary used for CLI/MCP/describe
- `input_schema`: JSON Schema object or path to a JSON Schema file

Optional fields:

- `output_schema`: JSON Schema object or path
- `runtime`: default `quickjs`; initial implementation accepts `quickjs`
- `sandbox`: `strict|fs|net`; default `strict`
- `permission_profile`: named profile that acts as a ceiling, never an escalation
- `timeout_ms`: tighter timeout than the active session/profile limit
- `packs`: MCP/describe pack membership; default `["custom"]`
- `secrets`: declared secret bindings available through the command context
- `read_only`: UX annotation only
- `destructive`: UX annotation only
- `expose`: controls whether the command is projected to CLI, MCP, and `describe`

Validation rules:

- Skill names follow the Agent Skills name rules and should match the parent directory.
- Command IDs use lowercase letters, numbers, and hyphens.
- Projected tool names are derived deterministically as `<skill>_<command>` with hyphens converted to underscores.
- Command names must not collide with built-in tool names or reserved meta-tools.
- Discovery never executes command code.
- Unknown Vulcan command metadata keys should be rejected or diagnosed so discovery stays deterministic.
- `sandbox = none` is not valid for projected skill commands.
- `permission_profile` can narrow authority but cannot widen the caller's authority.

## Runtime contract

The declared command runs in a fresh QuickJS context per invocation.

Entrypoint shape:

```js
export default function main(ctx, input) {
  return { result: input };
}
```

The runtime may also support CommonJS-style or global `main` compatibility where practical, but the documented convention is an exported default function.

Available context:

- `ctx.skill`: skill metadata and resolved paths
- `ctx.command`: command metadata and resolved paths
- `ctx.call`: invocation metadata such as caller surface and timestamp
- `ctx.secrets.list()`: list declared secret names
- `ctx.secrets.get(name)`: resolve a declared secret or return `null`
- `ctx.secrets.require(name)`: resolve a declared secret or throw
- `vault`, `web`, `tools`, and `help()` exactly as permitted by the effective sandbox/profile
- `host.exec()` only when the effective profile allows execution
- `host.shell()` only when the effective profile explicitly allows shell execution

Return contract:

- Any JSON-serializable value becomes the command result.
- A command may return `{ result, text }` to provide both structured output and a concise human fallback.
- A command may return `{ proposals }` for side effects that require preview, diff, checkpoint, or approval before application.
- If `output_schema` is present, Vulcan validates the structured result after execution.
- Throwing an error produces a structured command failure.

Operational rules:

- `console.log()` is diagnostic only.
- Each invocation is isolated; no persistent JS heap is shared across calls.
- Recursive tool/command calls require a depth cap and cycle detection.
- Structured data goes to stdout when commands are executed via shebang/shell compatibility; diagnostics go to stderr.

## Shebang and external harness compatibility

A command script should be executable by other harnesses that can run shell commands.

Recommended shebang:

```js
#!/usr/bin/env -S vulcan skill exec
```

The script should support:

```bash
scripts/prepare-day.js --help
scripts/prepare-day.js --json '{"date":"2026-05-05","dryRun":true}'
vulcan skill run daily-review prepare-day --input-json '{"date":"2026-05-05"}'
```

Scripts should be non-interactive, provide concise `--help`, use structured output, send diagnostics to stderr, support dry-run for mutating operations, and produce predictable output sizes.

## Secret integration

Skill commands declare secret bindings in `metadata.vulcan.commands.<command>.secrets`.

Example:

```yaml
metadata:
  vulcan:
    commands:
      github-status:
        script: scripts/github-status.js
        input_schema:
          type: object
          additionalProperties: false
          properties:
            repo: { type: string }
          required: [repo]
        secrets:
          - name: github_token
            ref: secret://github/personal/read-only
            required: true
            description: Token used for authenticated GitHub API calls
```

Rules:

- Secret values never live in `SKILL.md`.
- Discovery exposes binding metadata, not secret values.
- Missing required secrets fail at invocation time with a structured error.
- The command source uses `ctx.secrets.*`; backing storage can evolve without changing command code.
- Prefer brokered integration APIs over raw secret access whenever Vulcan has a typed integration wrapper.

## Shared registry and surface exposure

Skill commands plug into the same internal registry shape as built-in tools.

### CLI

Add or keep a stable skill command group:

- `vulcan skill list`
- `vulcan skill show <skill>`
- `vulcan skill commands <skill>`
- `vulcan skill run <skill> <command> --input-json <json>`
- `vulcan skill validate [<skill>]`
- `vulcan skill init <name>`

Projected commands may also appear under the generic tool surface:

- `vulcan tool list`
- `vulcan tool show <projected-name>`
- `vulcan tool run <projected-name> --input-json <json>`

`tool show` and `skill commands` should make clear that the projected tool comes from a skill command.

### Describe / external runtimes

`vulcan describe --format openai-tools|mcp|json-schema` includes visible projected skill commands from the shared registry. External runtimes should not need a separate adapter path for user-defined commands.

Discovery rule:

- Built-ins and projected skill commands share one registry.
- Visibility depends on trust, permission profile, selected packs, and the command's `expose` settings.
- Skill instructions remain readable through `skill show`, activation resources, and MCP resources.

### MCP

Projected skill commands are exposed as first-class MCP tools, not only through a generic `skill_run_command` escape hatch.

Required behavior:

- projected commands live in the same live registry as built-ins
- the default pack membership is `custom`
- changes under `.agents/skills` trigger `notifications/tools/list_changed`
- add resources:
  - `vulcan://assistant/skills/index`
  - `vulcan://assistant/skills/{name}`
  - `vulcan://assistant/skills/{name}/commands`
  - `vulcan://assistant/skills/{name}/resources`

The generic meta-tools remain useful:

- `skill_list`
- `skill_activate`
- `skill_run_command`

But projected commands provide better agent ergonomics.

### Internal JS API

Expose a `tools` namespace and a `skills` namespace in the shared JS runtime used by `vulcan run`, query JS, skill commands, and plugin hooks:

- `skills.list()`
- `skills.get(name)`
- `skills.commands(name)`
- `skills.activate(name)`
- `skills.run(name, command, input, opts?)`
- `tools.list()`
- `tools.get(name)`
- `tools.call(name, input, opts?)`

Nested calls preserve the current effective permission ceiling.

## Permissions and trust

Skill commands are executable vault code and require the same trust discipline as plugins.

Trust rules:

- Untrusted vaults may list skill metadata for local inspection if useful.
- Untrusted vault skill commands are not callable and are not exposed as live MCP/assistant tools.
- Executing a skill command requires a trusted vault.

Effective authority is the intersection of:

1. the active caller/session permission profile
2. the skill command's optional `permission_profile`
3. the command's declared sandbox ceiling
4. Vulcan's normal path/network/config/git/execute checks

A skill command can narrow authority; it cannot widen it.

Important rules:

- `read_only` and `destructive` are annotations, not authorization.
- Command code must use Vulcan APIs for vault mutation rather than editing files behind Vulcan's back.
- If a command requests a profile that does not exist, discovery marks the command invalid and it is not callable.
- Grants should be bound to the tuple `(skill_id, command_id, script_hash, permission_profile)` so code changes invalidate stale trust.

## Host execution

Provide two distinct APIs:

- `host.exec(argv, opts?)`
  - requires `execute`
  - no shell parsing
  - explicit argument vector only
- `host.shell(command, opts?)`
  - requires `shell`
  - opt-in and higher risk

Defaults:

- prefer `host.exec()` everywhere
- keep `host.shell()` off by default in recommended agent-facing profiles
- both APIs inherit timeout and output capture limits from the invocation unless a tighter limit is provided

## Relationship to skills

Skill commands are part of skills. The skill package is the portable unit; the command is the callable entrypoint.

Use a skill when the reusable asset includes knowledge, workflow guidance, examples, references, or executable helpers.

Declare a skill command when part of that skill should be callable by name with structured input and output.

## Relationship to plugins

Do not merge commands and plugins.

- Plugins subscribe to lifecycle events.
- Skill commands are invoked directly.
- Both reuse the same QuickJS runtime, trust checks, and permission machinery where practical.

Use a plugin when execution is triggered by a lifecycle event such as note write, scan completion, or pre-commit.

Use a skill command when execution is directly requested by a human, LLM, CLI command, MCP client, scheduler, or another script.

## Relationship to `vulcan run`

Use `vulcan run` for exploratory local scripts. Promote a script into a skill command when:

- it should be discovered by agents or humans
- it needs stable input/output schemas
- it needs permission and secret metadata
- it should be exported through CLI, MCP, `describe`, or JS APIs

## Future runtime slot: WASM

The registry should not hard-code QuickJS forever. Command metadata includes `runtime`, but the first implementation accepts only `quickjs`.

Future direction:

- support `runtime = "wasm"` under the same discovery and permission model
- keep `input_schema`, `output_schema`, trust, and pack exposure unchanged
- treat runtime choice as an execution backend detail, not a new surface

## Recommended implementation order

1. Build Agent Skills discovery for `.agents/skills/*/SKILL.md`.
2. Parse `metadata.vulcan.commands` into projected command definitions.
3. Add `vulcan skill list|show|commands|run|validate|init`.
4. Project skill commands into the shared internal tool registry.
5. Add static export through `describe --format openai-tools|mcp|json-schema`.
6. Add live MCP exposure and resource notifications.
7. Add the internal JS `skills.*` and `tools.*` namespaces.
8. Add `host.exec()` / `host.shell()` behind permissions.
9. Add scaffolding and docs polish once the contract is stable.
