# Vault-Native Custom Tools

This document specifies a first-class custom tool system for Vulcan. The goal is to let a vault
define reusable request/response tools that are callable by humans, external LLM runtimes, MCP
clients, and JavaScript running inside Vulcan without introducing a second extension runtime.

The initial implementation target is JavaScript-backed tools running in the existing QuickJS
runtime. The registry and manifest shape should be runtime-agnostic enough that a later WASM engine
can slot in under the same tool contract without redefining discovery, permissions, or surface
exposure.

## Goals

- Keep tools vault-native: a tool is stored in the vault, reviewed like any other note or script
  asset, and shipped alongside prompts and skills.
- Reuse the existing JS runtime, trust model, and permission profiles rather than inventing a second
  scripting subsystem.
- Make tool discovery static and cheap: list and describe tools from manifest data only, with no
  code execution during discovery.
- Keep the tool registry shared across CLI, `describe`, MCP, and internal JS APIs so the exported
  schema and the live callable surface do not drift.
- Preserve the current rule that vault mutations go through Vulcan semantics (`vault.*`, core
  commands, permission checks, diagnostics), even when the caller is a custom tool.
- Support host process execution deliberately and explicitly rather than smuggling it behind
  `sandbox = none`.

## Non-goals for the initial version

- A second plugin/event system. Event-driven plugins and callable tools remain separate concepts.
- Dynamic tool schemas generated by running user code at discovery time.
- Long-lived in-memory tool daemons or background workers.
- Arbitrary runtime loading from npm, pip, or remote URLs.
- Full WASM runtime support in the first implementation.

## Core model

Custom tools are callable assets, not hooks:

- Skills teach the model or human how to work.
- Plugins react to Vulcan lifecycle events.
- Custom tools accept structured input and return structured output on demand.

If a JS asset is meant to be reused as a direct callable function across CLI, MCP, external
assistants, and the internal JS API, it should be a custom tool rather than a script hidden inside a
skill directory.

## Tool asset layout

Tool files live under a configurable folder:

- `assistant.tools_folder` in `.vulcan/config.toml`
- Default: `.agents/tools`

Directory layout:

```text
.agents/tools/
  summarize_meeting/
    TOOL.md
    main.js
```

Initial rules:

- One directory per tool.
- `TOOL.md` is required and contains static metadata plus human-readable documentation.
- `main.js` is the default entrypoint.
- Additional files may exist in the tool directory, but Phase 1 only guarantees execution of the
  declared entrypoint. Local module/import ergonomics can be improved later without changing the
  manifest contract.

## Manifest format

`TOOL.md` is Markdown with YAML frontmatter. Discovery uses only the frontmatter and file path; the
body is for human/LLM guidance and examples.

Required fields:

- `name`: canonical tool identifier, `snake_case`, globally unique across built-in and custom tools
- `description`: one-line summary
- `input_schema`: JSON Schema object describing accepted input

Optional fields:

- `title`
- `version`
- `entrypoint` ã default `main.js`
- `runtime` ã default `quickjs`; Phase 1 only accepts `quickjs`
- `tags`
- `sandbox` ã `strict|fs|net`; default `strict`
- `permission_profile` ã optional named profile that acts as a ceiling, never an escalation
- `timeout_ms` ã optional tighter execution timeout than the active session/profile limit
- `packs` ã optional MCP/describe pack membership; default `["custom"]`
- `secrets` ã optional list of named secret bindings exposed through `ctx.secrets`
- `read_only` ã hint for UX/annotations only
- `destructive` ã hint for UX/annotations only
- `output_schema` ã JSON Schema object for the tool result payload

Initial validation rules:

- Names must not collide with built-in tool names, other custom tools, or reserved meta-tools such
  as `tool_list`, `tool_show`, `tool_run`.
- Unknown manifest keys should be rejected so discovery stays deterministic.
- `sandbox = none` is not allowed for custom tools. Resource limits remain active for all custom
  tool execution.
- Pack names must be canonical registered pack names plus `custom`; unknown pack names make the tool
  invalid.
- Secret declarations must use unique `snake_case` names and valid environment variable names.

Example:

```markdown
---
name: summarize_meeting
title: Summarize Meeting Note
description: Read one meeting note and return decisions, action items, and follow-ups.
version: 1
runtime: quickjs
entrypoint: main.js
tags: [meetings, summaries]
sandbox: fs
permission_profile: readonly
timeout_ms: 5000
packs: [custom]
secrets:
  - name: openai_api_key
    env: OPENAI_API_KEY
    required: true
    description: API key used for the remote summarization call
read_only: true
input_schema:
  type: object
  additionalProperties: false
  properties:
    note:
      type: string
      description: Path or resolvable note name
  required: [note]
output_schema:
  type: object
  additionalProperties: false
  properties:
    title:
      type: string
    decisions:
      type: array
      items: { type: string }
    action_items:
      type: array
      items:
        type: object
        additionalProperties: false
        properties:
          text: { type: string }
          owner: { type: string }
        required: [text]
  required: [title, decisions, action_items]
---

## When to use

Use this tool when you already know the note to summarize and want a structured result rather than a
free-form assistant answer.
```

## Runtime contract

The declared entrypoint runs in a fresh QuickJS context per invocation.

Entrypoint shape:

```js
function main(input, ctx) {
  // return JSON-serializable data or a structured envelope
}
```

Available values inside the runtime:

- `input`: validated JSON object/value matching `input_schema`
- `ctx.tool`: manifest metadata plus resolved paths
- `ctx.call`: invocation metadata such as tool name, timestamp, and caller surface
- `ctx.secrets.list()`: list declared secret names
- `ctx.secrets.get(name)`: resolve a declared secret or return `null`
- `ctx.secrets.require(name)`: resolve a declared secret or throw if unavailable
- `vault`, `web`, and `help()` exactly as permitted by the effective sandbox/profile
- `tools.*` for calling other tools from JS
- `host.*` only when the effective profile allows execution

Return contract:

- If `main()` returns any JSON-serializable value, Vulcan treats it as the tool `result`.
- A tool may instead return `{ result, text }` to provide both structured output and a concise text
  fallback for surfaces such as MCP.
- If `output_schema` is present, it validates `result` after execution.
- Throwing an error produces a structured tool failure.

Operational rules:

- `console.log()` is diagnostic only; tools should communicate through return values.
- Each invocation is isolated. No persistent JS heap or module cache is shared across calls.
- Recursive tool calls are allowed only with an explicit depth cap and cycle detection.

## Secret integration

Phase 1 uses manifest-declared secret bindings backed by environment variables. `TOOL.md` declares
the secret name, the environment variable to read, whether it is required, and optional
documentation for humans:

```yaml
secrets:
  - name: github_token
    env: GITHUB_TOKEN
    required: true
    description: Token used for authenticated GitHub API calls
```

Rules:

- Secret values never live in `TOOL.md`; the manifest contains references only.
- Discovery exposes the binding metadata, not the secret values.
- Missing required secrets fail at invocation time with a structured error.
- Optional secrets resolve to `null` from `ctx.secrets.get(name)` when absent.
- The Phase 1 abstraction is `ctx.secrets.*`; backing storage can evolve later without changing tool
  source code.

## Shared registry and surface exposure

Custom tools should plug into the same internal registry shape as built-in tools.

### CLI

Add a stable `tool` command group:

- `vulcan tool list`
- `vulcan tool show <name>`
- `vulcan tool run <name> --input-json <json>`
- `vulcan tool init <name>`
- `vulcan tool set <name> ...`
- `vulcan tool validate [<name>]`

`tool show` returns parsed manifest metadata plus the documentation body. `tool run --output json`
returns a stable envelope containing the canonical tool name, validated input summary, result, and
optional text fallback. When no explicit input is provided, `tool run` uses `{}`.

Authoring helpers:

- `tool init` scaffolds a new tool directory with `TOOL.md` and `main.js`, plus optional starter
  examples. Phase 1 ships a minimal starter that already demonstrates `main(input, ctx)`,
  structured `{ result, text }` returns, and the `ctx.secrets.*` runtime helpers.
- `tool set` performs structured manifest edits so users do not need to hand-edit YAML frontmatter
  for common changes like description, sandbox, permission profile, packs, secret bindings, or
  schema file paths.
- `tool validate` checks manifest shape, schema validity, pack names, entrypoint existence, and
  permission-profile references for one tool or the whole tools folder.

Current `tool set` conventions:

- `--pack <name>` replaces the full pack list for the tool.
- `--secret name=ENV` replaces the full secret list for the tool.
- `--input-schema-file` and `--output-schema-file` read JSON schema objects from files.

Design rule: prefer deterministic scaffold/update commands over editor-launching commands. A future
interactive TUI/editor helper is acceptable, but non-interactive authoring commands should remain the
primary contract.

### Describe / external runtimes

`vulcan describe --format openai-tools|mcp|json-schema` should include visible custom tools from the
shared registry. External runtimes should not need a separate adapter path for user-defined tools.

Discovery rule:

- Built-ins and custom tools share one registry.
- Visibility still depends on trust, permission profile, and selected pack set.
- Tool documentation remains separately readable through `tool show` and MCP resources.

### MCP

Custom tools are exposed as first-class MCP tools, not through a generic `run_custom_tool` escape
hatch.

Required MCP behavior:

- custom tools live in the same live registry as built-ins
- the default pack membership is `custom`
- changes under `assistant.tools_folder` trigger `notifications/tools/list_changed`
- add resources:
  - `vulcan://assistant/tools/index`
  - `vulcan://assistant/tools/{name}`

### Internal JS API

Expose a `tools` namespace in the shared JS runtime used by `vulcan run`, `dataview query-js`,
custom tools, and plugin hooks:

- `tools.list()`
- `tools.get(name)`
- `tools.call(name, input, opts?)`

This lets scripts, plugins, and future built-in assistant flows compose reusable tool logic instead
of duplicating it.

Runtime behavior:

- `tools.list()` returns visible custom tools with static metadata and `callable`.
- `tools.get(name)` returns one tool definition plus its Markdown body.
- `tools.call(name, input, opts?)` validates input, runs the target tool, and returns either the
  plain JSON result or `{ result, text }` when the callee provided a human fallback.
- Nested calls preserve the current effective permission ceiling.
- Recursive call loops are rejected and nested tool composition is capped at depth 8.

## Embedded help and in-repo docs

This feature should ship with first-class documentation in both the integrated help surface and the
repository docs. Documentation is part of the product surface, not follow-up polish.

Required embedded help topics:

- `help tool`
- `help tool init`
- `help tool set`
- `help tool validate`
- `help js.tools`
- `help js.host`
- `help js.plugins`
- one comparison-oriented topic explaining when to use scripts vs skills vs tools vs plugins

Required in-repo docs:

- a user-facing guide comparing automation surfaces with examples
- a reference doc for tool authoring and manifest fields
- a plugin doc that explicitly contrasts event hooks with callable custom tools and Markdown skills
- end-to-end examples for at least:
  - a read-only analysis tool
  - a write-capable vault mutation tool
  - a host-exec tool guarded by permissions
  - a plugin that should remain a plugin rather than becoming a tool

Documentation quality bar:

- include examples, not just schema prose
- explain why one surface is appropriate and another is not
- keep embedded help and repo docs generated from or at least anchored to the same canonical field
  names and command shapes where possible

## Permissions and trust

Custom tools are executable vault code and therefore require the same trust discipline as plugins.

Trust rules:

- untrusted vaults may list custom tool metadata in local CLI inspection if useful, but custom tools
  are not callable and are not exposed as live MCP/assistant tools
- executing a custom tool requires a trusted vault

Effective authority is the intersection of:

1. the active caller/session permission profile
2. the tool's optional `permission_profile`
3. the tool's declared sandbox ceiling
4. Vulcan's normal path/network/config/git/execute checks

A custom tool can narrow authority; it cannot widen it.

Important rules:

- `read_only` and `destructive` are annotations, not authorization.
- Tool code must still use Vulcan APIs for vault mutation rather than editing files behind Vulcan's
  back.
- If a tool requests a profile that does not exist, discovery marks the tool invalid and it is not
  callable.

## Host execution

Host execution is the part that most needs crisp boundaries.

Provide two distinct APIs:

- `host.exec(argv, opts?)`
  - requires `execute`
  - no shell parsing
  - explicit argument vector only
- `host.shell(command, opts?)`
  - requires `shell`
  - opt-in and higher risk

Defaults:

- prefer `host.exec()` everywhere
- keep `host.shell()` off by default in recommended agent-facing profiles
- both APIs inherit timeout and output capture limits from the tool invocation unless a tighter limit
  is provided

Phase 1 can rely on the existing `execute`/`shell` grants plus trusted-vault gating. If Vulcan
starts exposing these tools broadly in daemon or multi-user contexts, command allowlists should be
added before widening the default profiles.

## Relationship to skills

Skills remain Markdown guidance. They can mention custom tools by name exactly like built-in tools.

Recommended rule:

- use a skill when the reusable asset is knowledge, workflow guidance, heuristics, or examples
- use a custom tool when the reusable asset is a callable function with structured input/output

Advanced skill-local scripts remain valid for harness-specific workflows, but if the script should be
discoverable and callable through CLI, MCP, `describe`, and the JS API, it should be promoted into
`.agents/tools`.

## Relationship to plugins

Do not merge tools and plugins.

- plugins subscribe to lifecycle events
- tools are invoked directly
- both reuse the same QuickJS runtime, trust checks, and permission machinery where practical

This separation keeps "policy hook" or "pre-commit linter" logic distinct from "summarize this note"
or "call an external service and write the result" logic.

Short comparison:

- Use a plugin when execution is triggered by a lifecycle event such as note write, scan completion,
  or pre-commit.
- Use a custom tool when execution is directly requested by a human, LLM, CLI command, MCP client,
  or another script and the interaction has a typed input/output contract.
- Use a skill when the asset is guidance, examples, heuristics, or workflow instructions rather than
  executable behavior.

## Future runtime slot: WASM

The registry should not hard-code QuickJS forever. The manifest already includes `runtime`, but Phase
1 accepts only `quickjs`.

Future direction:

- support `runtime = "wasm"` under the same discovery and permission model
- keep `input_schema`, `output_schema`, trust, and pack exposure unchanged
- treat runtime choice as an execution backend detail, not a new kind of tool surface

## Recommended implementation order

1. Build the manifest parser and loader for `.agents/tools`.
2. Add a shared internal tool registry that can contain both built-ins and custom entries.
3. Add `vulcan tool list|show|run`.
4. Add static export through `describe --format openai-tools|mcp|json-schema`.
5. Add live MCP exposure and resource notifications.
6. Add the internal JS `tools.*` namespace.
7. Add `host.exec()` / `host.shell()` behind permissions.
8. Add scaffolding and docs polish once the contract is stable.
